# Тема 4.2: UDP (User Datagram Protocol)

Изучение протокола UDP, его структуры, областей применения (реальное время, DNS, NTP, групповой трафик), а также сравнения с TCP и принципов работы.

---

## 1. UDP — Базовые свойства

- **Уровень OSI:** Транспортный уровень (L4).
- **Сущность данных (PDU):** Датаграмма.
- **Принцип работы:** UDP (User Datagram Protocol) — протокол, описанный в **RFC 768** (https://www.rfc-editor.org/rfc/rfc768), обеспечивает передачу данных между приложениями без гарантии доставки.

### Ключевые особенности

| **Особенность** | **Описание** |
| :--- | :--- |
| **Ненадёжность** | UDP не заботится о доставке данных. Пакет может потеряться или прийти в другом порядке. |
| **Нет контроля перегрузок** | UDP может даже не отправить пакет, если буфер на отправку переполнен. |
| **Малый заголовок** | Всего **8 байт** (против 20+ у TCP). |
| **Нет установки соединения** | Отправка происходит без предварительного «рукопожатия». |
*Таб. 1. Ключевые особенности протокола UDP.*

### Зачем нужен UDP, если он ненадёжный?

UDP отлично решает задачи, для которых он был создан:

1. **Передача трафика в реальном времени** (видеоконференции, онлайн-игры).
2. **Скорость важнее надёжности** (DNS, NTP).
3. **Передача группового (Multicast) трафика.**

> **Шутка для запоминания:** *«Я знаю отличную шутку про UDP, но не факт, что она до вас дойдёт»* — отлично иллюстрирует ненадёжность протокола.

## 2. Структура UDP-заголовка

UDP-заголовок занимает всего **8 байт** (минимальный размер для транспортного протокола).

| **Поле** | **Размер** | **Назначение** |
| :--- | :--- | :--- |
| `Source Port` | 2 байта | Порт источника (отправителя). |
| `Destination Port` | 2 байта | Порт назначения (получателя). |
| `Length` | 2 байта | Длина всей UDP-датаграммы в байтах (заголовок + данные). Минимум — 8 байт (без данных). |
| `Checksum` | 2 байта | Контрольная сумма для проверки целостности пакета. |
*Таб. 2. Структура UDP заголовка и его назначение.*

> **Важно:** Поле `Length` указывает общий размер UDP-датаграммы. Максимальный размер данных в одном UDP-пакете ограничен MTU (обычно 1472 байта для Ethernet).

## 3. Практические применения UDP

### 3.1. Передача данных в реальном времени

**Примеры:** Видеоконференции, онлайн-игры (CS, Dota 2, StarCraft).

**Почему UDP, а не TCP:**
- Если пакет теряется при использовании TCP, протокол пытается его восстановить — это вызывает задержки и «подвисания» игры. К тому же, восстановленные данные могут быть уже неактуальны, так как состояние сервера изменилось.
- При использовании UDP потеря пакета вызывает лишь кратковременный «лаг», но последующие пакеты содержат актуальное состояние сервера.

**Аналогия:** При видеоконференции потери и восстановления TCP пакетов привели бы к отставанию видео от реальности. UDP жертвует надёжностью ради минимальной задержки.

### 3.2. Скорость важнее надёжности

**Примеры:** DNS, NTP.

| **Протокол** | **Почему используется UDP** |
| :--- | :--- |
| **DNS** | Один DNS-запрос через UDP = 2 пакета (запрос + ответ). При потере — повторный запрос. Через TCP один запрос = 9 пакетов, что в разы неэффективнее. |
| **NTP** | Для синхронизации времени критично точно измерять задержки. TCP с повторными передачами исказил бы расчёты. |
*Таб. 3. Примеры почему некоторые протоколы используют UDP.*

>**Почему TCP использует 9 пакетов для одного DNS-запроса?**
>В отличие от UDP, TCP перед передачей данных выполняет **тройное рукопожатие (3-way handshake)** для установки соединения (3 пакета), а затем подтверждает получение каждого пакета (ACK) и корректно закрывает соединение. В сумме это даёт около 9 служебных пакетов, что крайне неэффективно для коротких запросов. UDP же просто отправляет запрос и получает ответ в рамках 2 пакетов, не тратя время на служебные процедуры.

### 3.3. Передача группового трафика (Multicast)

UDP — **единственный** транспортный протокол в стеке TCP/IP, поддерживающий групповую рассылку.

**Пример:** «Разливка» образов ОС на 50 компьютеров. Отправлять десятки гигабайт каждому хосту через TCP (Unicast) — слишком долго и затратно. Использование UDP + Multicast позволяет отправить данные **один раз** всем получателям в группе.

*(Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=4fc0fafb-2a16-4244-a664-3f1e8f788a63))* - групповой трафик (мультикаст).

> **Важный нюанс:** Хотя UDP не гарантирует доставку, на уровне приложения можно реализовать свои механизмы надёжности (например, нумерация пакетов и ретрансляция потерянных, как делают утилиты «разливки» образов).

## 4. Разбор сложных инженерных ситуаций и аномалий

### 4.1. Контрольная сумма в UDP

**Вопрос:** *Зачем в UDP контрольная сумма, если протокол ненадёжный?*

**Технический ответ:** Контрольная сумма нужна для **обнаружения битовых ошибок**, возникших при передаче (например, из-за помех в кабеле). Раньше сети были менее надёжны, и это позволяло отбрасывать повреждённые пакеты, не передавая их наверх. Если контрольная сумма не совпадает — пакет **молча уничтожается**.

### 4.2. Размер UDP-пакета

**Вопрос:** *Какой максимальный размер данных может быть в UDP-пакете?*

**Технический ответ:** Теоретически, поле `Length` позволяет передать до `65535` байт (вместе с UDP-заголовком). На практике размер ограничен **MTU** канального уровня:
- Ethernet MTU = 1500 байт.
- Вычитаем IP-заголовок (20 байт) → 1480.
- Вычитаем UDP-заголовок (8 байт) → **1472 байта данных** в одном пакете.

### 4.3. Различия между TCP и UDP

| **Критерий**            | **TCP**                         | **UDP**                          |
| :---------------------- | :------------------------------ | :------------------------------- |
| **Надёжность**          | Гарантирует доставку            | Не гарантирует                   |
| **Скорость**            | Медленнее (из-за проверок)      | Быстрее                          |
| **Соединение**          | Устанавливает (3-way handshake) | Не устанавливает                 |
| **Контроль перегрузок** | Есть                            | Нет                              |
| **Повторная передача**  | Есть                            | Нет                              |
| **Multicast**           | Не поддерживает                 | Поддерживает                     |
| **Заголовок**           | 20+ байт                        | **8 байт**                       |
| **Примеры**             | Веб, почта, файлообмен          | Видео, игры, DNS, NTP, Multicast |
*Таб. 4. Различия по критериям между TCP и UDP.*

## 5. Расширенный ИБ-контекст и инженерный FAQ

### 1. Векторы атак и уязвимости

- **UDP Flood (DoS/DDoS):** Злоумышленник отправляет огромное количество UDP-пакетов на случайные порты, перегружая целевой хост или сеть.
- **DNS Amplification:** Использование UDP для усиления DDoS-атак (подмена Source IP на IP жертвы, отправка запросов к DNS-серверам с большим ответом).
- **UDP Spoofing:** Подмена IP-адреса отправителя в UDP-пакетах для обхода фильтрации или скрытия источника атаки.
- **UDP-сканирование:** Злоумышленники сканируют открытые UDP-порты для поиска уязвимых служб (SNMP, DNS, NTP).

### 2. Инженерный FAQ (На основе разбора дискуссий)

- **Вопрос:** *Какой протокол транспортного уровня используется для групповой рассылки?*
- **Технический ответ:** **UDP**, так как он единственный в стеке TCP/IP поддерживает Multicast.

- **Вопрос:** *Какова длина UDP-заголовка?*
- **Технический ответ:** **8 байт**.

- **Вопрос:** *Почему UDP лучше TCP для DNS-запросов?*
- **Технический ответ:** UDP требует минимального количества пакетов (1 запрос + 1 ответ). TCP для установки соединения требует 3 пакетов (рукопожатие) плюс передача данных и закрытие, что неэффективно для коротких запросов.

- **Вопрос:** *Что происходит с UDP-пакетом, если контрольная сумма не совпадает?*
- **Технический ответ:** Пакет **молча уничтожается**. Отправитель не получает никакого уведомления об ошибке.

- **Вопрос:** *Если UDP не восстанавливает пакеты, как тогда работают «разливки» образов через Multicast?*
- **Технический ответ:** Надёжность обеспечивается на **уровне приложения**. Например, утилита может нумеровать пакеты и ретранслировать потерянные по запросу клиента. В отличие от TCP, это позволяет эффективно использовать групповую рассылку.

---

## Заключение

UDP — это лёгкий и быстрый транспортный протокол, идеально подходящий для сценариев, где скорость важнее надёжности: потоковое видео, онлайн-игры, DNS, NTP и групповая рассылка (Multicast). Его 8-байтовый заголовок обеспечивает минимальные накладные расходы, а отсутствие механизмов восстановления данных возлагает ответственность за целостность и порядок на прикладной уровень. Понимание различий между TCP и UDP является фундаментом для выбора правильного протокола при проектировании сетевых приложений.
