# Тема 4.3: TCP (Transmission Control Protocol)

Изучение структуры, механизмов и принципов работы протокола TCP, включая установку соединения (3-way handshake), передачу данных с подтверждением, обнаружение потерь и восстановление, управление скоростью, а также диагностику состояний соединения.

---

## 1. TCP — Базовые свойства

- **Уровень OSI:** Транспортный уровень (L4).
- **Сущность данных (PDU):** Сегмент.
- **Принцип работы:** TCP (Transmission Control Protocol) — протокол с **гарантией** доставки данных, обеспечивающий надёжную передачу потока байт между приложениями. Впервые описан в [**RFC 793**](https://datatracker.ietf.org/doc/html/rfc793) (1981 год).

TCP является **полной противоположностью** UDP:

| **Критерий**            | **TCP**                         | **UDP**          |
| :---------------------- | :------------------------------ | :--------------- |
| **Надёжность**          | Гарантирует доставку            | Не гарантирует   |
| **Соединение**          | Устанавливает (3-way handshake) | Не устанавливает |
| **Контроль перегрузок** | Есть                            | Нет              |
| **Повторная передача**  | Есть                            | Нет              |
| **Multicast**           | Не поддерживает                 | Поддерживает     |
| **Заголовок**           | 20+ байт                        | 8 байт           |
| **Тип данных**          | Поток байт                      | Датаграммы       |
*Таб. 1. Различия по критериям между TCP и UDP.*

## 2. Структура TCP-заголовка

| **Поле** | **Размер** | **Назначение** |
| :--- | :--- | :--- |
| `Source Port` | 2 байта | Порт источника (отправителя). |
| `Destination Port` | 2 байта | Порт назначения (получателя). |
| `Sequence Number` | 4 байта | Номер последовательности — указывает позицию данных в буфере отправителя. |
| `Acknowledgment Number` | 4 байта | Номер подтверждения — указывает, какие данные успешно получены. |
| `Data Offset (IHL)` | 4 бита | Длина TCP-заголовка в 32-битных словах. |
| `Flags` | 9 бит | Управляющие флаги: `URG`, `ACK`, `PSH`, `RST`, `SYN`, `FIN`, `ECE`, `CWR`, `NS`. |
| `Window` | 2 байта | Размер окна — количество данных, которое отправитель готов принять. |
| `Checksum` | 2 байта | Контрольная сумма для проверки целостности сегмента. |
| `Urgent Pointer` | 2 байта | Указатель на срочные данные (при флаге URG). |
| `Options` | переменная | Дополнительные опции (например, MSS, SACK, Window Scaling). |
*Таб. 2. Структура полей TCP-заголовка ( их размер и назначение ).*

**Стандартный размер TCP-заголовка без опций — 20 байт.**

### Расшифровка флагов TCP

| **Флаг** | **Название** | **Назначение** |
| :--- | :--- | :--- |
| `SYN` | Synchronize | Используется для установки соединения (3-way handshake). |
| `ACK` | Acknowledgment | Подтверждение успешного получения данных. |
| `FIN` | Finish | Завершение соединения. |
| `RST` | Reset | Сброс соединения (отказ от установки или аварийное завершение). |
| `PSH` | Push | Немедленная передача данных приложению (без буферизации). |
| `URG` | Urgent | Указание на срочные данные в сегменте. |
| `ECE` | ECN Echo | Индикация поддержки ECN (управление перегрузками). |
| `CWR` | Congestion Window Reduced | Уменьшение окна перегрузки. |
| `NS` | Nonce Sum | Экспериментальный флаг для защиты от скрытия пакетов. |
*Таб. 3. Расшифровка флагов TCP ( их название и назначение ).*

## 3. Установка TCP-соединения (3-way Handshake)

Любая передача данных в TCP начинается с установки **виртуального соединения**. Это необходимо для проверки готовности удалённого приложения к обмену данными.

**Аналогия:** Установка соединения — это как звонок по телефону. Если пользователь поднимает трубку, можно начинать общение. Если нет — нет смысла ничего говорить в трубку.

### Трёхэтапное рукопожатие

| **Шаг** | **Отправитель** | **Направление** | **Получатель** | **Содержание пакета** | **Описание** |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | **Клиент** | → | Сервер | `SYN (Seq=x)` | Клиент инициирует соединение, отправляя SYN-пакет со случайным номером последовательности. |
| 2 | **Сервер** | ← | Клиент | `SYN+ACK (Seq=y, Ack=x+1)` | Сервер подтверждает получение SYN и отправляет свой SYN-пакет с подтверждением. |
| 3 | **Клиент** | → | Сервер | `ACK (Seq=x+1, Ack=y+1)` | Клиент подтверждает получение SYN+ACK. Соединение установлено. |
*Таб. 4. Трёхэтапное рукопожатие TCP (3-way handshake).*

**Итого:** 3 пакета для установки соединения (ARP-запросы не учитываются).

> **Почему `Seq+1` и `Ack+1`?**
> TCP работает с **потоком байт**, а не с пакетами. Пакет с флагом `SYN` (как и `FIN`) занимает в этом потоке **ровно 1 байт**. Поэтому при подтверждении получения SYN, получатель всегда указывает в поле `Ack` следующее значение: `Seq + 1`. Это означает: «Я получил данные до этого байта включительно и ожидаю следующий байт».

## 4. Сценарии отказа при установке соединения

| **Сценарий** | **Ответ сервера** | **Поведение клиента** | **Итог** |
| :--- | :--- | :--- | :--- |
| **Порт закрыт** (нет приложения) | `RST` (Reset) | Немедленное закрытие соединения | Ошибка: "Connection refused" |
| **Хост не отвечает** (выключен / фаервол) | Нет ответа (таймаут) | Повторные попытки с увеличением интервала (1с, 2с, 4с, 8с...) | Ошибка: "Connection timed out" |
| **Хост недоступен** (нет маршрута) | `ICMP Destination Unreachable` | Остановка попыток | Ошибка: "Network unreachable" |
| **Переполнение очереди SYN** (SYN Flood) | Нет ответа | Повторные попытки (если очередь заполнена) | Отказ в обслуживании (DoS) |
*Таб. 5. Сценарии отказа при установке TCP-соединения.*

### 4.1. Порт закрыт (приложение не слушает)

Если на сервере нет приложения, готового принимать данные на указанном порту, в ответ на SYN приходит пакет с флагом **RST** (Reset).

*Попытка установки TCP соединения на закрытый порт - (Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=b768b8a6-322e-4520-8a7a-1da02b4cd008))*

> **Важно:** Флаг RST говорит клиенту, что соединение не может быть установлено, и попытки повторять не нужно.

### 4.2. Нет ответа (хост не отвечает или фаервол)

Если на SYN-пакет не приходит ни SYN+ACK, ни RST, TCP считает, что пакет потерялся, и начинает повторные попытки.

*Попытка установки TCP соединения на хост, который не отвечает - (Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=c16763e6-1ce8-4d50-b3b8-4261ddc9940b))*

**Повторные попытки (Exponential Backoff):**
1-я повторная попытка — через 1 секунду.
2-я — через 2 секунды.
3-я — через 4 секунды.
4-я — через 8 секунд.

> **Зачем нужен Exponential Backoff?**
> Экспоненциальное увеличение задержки позволяет распределить нагрузку на сеть: если сервер временно недоступен или только поднимается, повторные попытки не создадут лавину запросов от множества клиентов одновременно.

> **Важно:** Количество попыток ограничено настройками ОС (обычно 4). Именно поэтому приложения «зависают» при попытке подключения к недоступному хосту.

## 5. Диаграмма состояний TCP

TCP — **протокол с состояниями**. В отличие от UDP, каждое TCP-соединение проходит через несколько состояний:

| **Состояние**  | **Описание**                                        |
| :------------- | :-------------------------------------------------- |
| `CLOSED`       | Начальное состояние.                                |
| `LISTEN`       | Ожидание входящего соединения (сервер).             |
| `SYN-SENT`     | Отправлен SYN, ожидание SYN+ACK.                    |
| `SYN-RECEIVED` | Получен SYN, отправлен SYN+ACK.                     |
| `ESTABLISHED`  | Соединение установлено, идёт передача данных.       |
| `FIN-WAIT-1`   | Отправлен FIN, ожидание ACK.                        |
| `FIN-WAIT-2`   | Получен ACK на FIN, ожидание FIN от другой стороны. |
| `TIME-WAIT`    | Ожидание исчезновения старых пакетов (2 MSL).       |
| `CLOSE-WAIT`   | Получен FIN, ожидание закрытия приложения.          |
| `LAST-ACK`     | Отправлен FIN, ожидание последнего ACK.             |
| `CLOSING`      | Обе стороны отправили FIN одновременно.             |
*Таб. 6. Состояния TCP-соединения.*

> **Примечание:** Завершение соединения (4-way handshake): FIN+ACK → ACK → FIN+ACK → ACK.

>**Почему TIME-WAIT длится так долго?**  
>Состояние `TIME-WAIT` длится **2 MSL** (Maximum Segment Lifetime, обычно ~60 секунд). Это нужно, чтобы все старые, запоздавшие пакеты этого соединения гарантированно «умерли» в сети и случайно не попали в новое соединение с теми же портами (защита от перемешивания данных).

## 6. Передача данных: Sequence и Acknowledgment Numbers

### 6.1. Sequence Number (Номер последовательности)

TCP — это **потоковый** протокол. Он передаёт данные как непрерывный поток байт, а не отдельные датаграммы (как UDP).

**Проблема:** Пакеты могут прийти в неправильном порядке.

**Решение:** Использовать `Sequence Number` — номер, указывающий позицию данных в буфере отправителя.

**Пример:** Отправляем 2800 байт в 3 пакетах:

| **Пакет** | **Размер данных** | **Sequence Number** |
| :--- | :--- | :--- |
| №1 | 1000 байт | `0` |
| №2 | 800 байт | `1000` |
| №3 | 1000 байт | `1800` |
*Таб. 7. Пример использования Sequence Number (2800 байт в 3 пакетах).*

Даже если пакеты придут в порядке 3 → 1 → 2, принимающая сторона разместит их правильно по полю `Sequence Number`.

### 6.2. Acknowledgment Number (Номер подтверждения)

Используется для подтверждения успешного получения данных.

**Принцип:** `Acknowledgment Number` = последний полученный байт + 1 (следующий ожидаемый байт).

**Пример:**
1. Отправлен пакет: `Seq=0`, размер = 1000.
2. Получен ACK: `Ack=1000` (означает, что байты 0–999 получены, ожидается байт 1000).

### 6.3. Начальное значение Sequence Number

Начальный номер последовательности (ISN) выбирается **случайно** при установке соединения. Это делается для безопасности (защита от TCP-hijacking).

## 7. Обнаружение потерь и повторная передача

Если отправитель не получает ACK в течение определённого времени (таймаут), пакет считается потерянным и отправляется повторно.

**Важно:** TCP не знает, что именно произошло — потерялся пакет с данными или потерялся ACK. В обоих случаях он повторно отправляет данные.

> **Современное улучшение:** SACK (Selective Acknowledgment) — опция, позволяющая подтверждать выборочные пакеты, а не только непрерывную последовательность.

## 8. Управление скоростью передачи (Окно)

Передача одного пакета и ожидание ACK перед отправкой следующего — надёжно, но **медленно**.

**Решение:** Использовать **окно** (`Window`) — количество данных, которое можно отправить без подтверждения. Размер окна динамически регулируется в зависимости от состояния сети и загрузки получателя.

**Почему это так медленно?**
Представим, что клиент находится во Владивостоке, а сервер — в Калининграде (около 7500 км). Даже со скоростью света (~300 000 км/с) сигнал идёт туда и обратно ~0.1 секунды (RTT). В идеале за эту секунду мы можем отправить около 20 пакетов. 
Максимальный размер одного пакета с данными — 1460 байт (1500 MTU − 20 IP − 20 TCP). 
20 пакетов × 1460 байт = **28 900 байт/с (~28 Кбайт/с)**. Это скорость, с которой интернет работал в 90-х! Современные приложения не могут работать с такой скоростью.

Именно поэтому TCP использует **скользящее окно** — чтобы отправлять несколько пакетов без ожидания подтверждения, не снижая скорость.

## 9. Разбор сложных инженерных ситуаций и аномалий

### 9.1. Поток байт vs Датаграммы

**Вопрос:** *В чём отличие потока байт (TCP) от датаграмм (UDP)?*

**Технический ответ:** UDP передаёт данные как отдельные датаграммы — что отправили, то и дошло (если дошло). TCP же принимает от приложения поток байт и сам решает, как нарезать его на сегменты. Например, при отправке 3000 байт через UDP будет одна датаграмма (которая фрагментируется на IP-уровне). TCP разобьёт эти данные на несколько сегментов оптимального размера (например, 2×1460 + 80 байт).

### 9.2. Обработка повторных пакетов

**Вопрос:** *Что происходит, если отправитель повторно отправил пакет, но получатель уже получил оригинал?*

**Технический ответ:** Зависит от реализации. Получатель может отбросить дублирующие данные, но **всегда** отправляет подтверждение (`ACK`) в ответ на любой полученный сегмент.

> **Почему получатель отправляет ACK на дубликат?**
> Отправитель не знает, потерялся пакет с данными или пакет с подтверждением (ACK). Если ACK не приходит, отправитель считает, что потерялись данные, и отправляет их снова. Когда получатель получает дубликат, он понимает: клиент не получил мой ACK. Поэтому он снова отправляет ACK, чтобы сообщить: «Данные уже получены, не отправляй их больше». Без этого повторного ACK клиент продолжал бы бесконечно отправлять один и тот же пакет.

### 9.3. Закрытие TCP-соединения

Закрытие происходит в **4 этапа** (4-way handshake):
1. `FIN` → от одной стороны.
2. `ACK` ← от другой стороны.
3. `FIN` ← от другой стороны.
4. `ACK` → от первой стороны.

В некоторых случаях (например, в Miminet) FIN+ACK могут объединяться.

### 9.4. Почему в TCP нет «состояний» для UDP?

UDP не устанавливает соединение, поэтому у него нет состояний (за исключением искусственных `UNCONN` или `ESTABLISHED` в некоторых реализациях, если был вызван `connect()`).

## 10. Расширенный ИБ-контекст и инженерный FAQ

### 1. Векторы атак и уязвимости

- **SYN Flood (DDoS):** Злоумышленник отправляет множество SYN-пакетов с поддельным Source IP, заполняя очередь полуоткрытых соединений на сервере и делая его недоступным.
- **TCP Hijacking:** Злоумышленник пытается захватить уже установленное TCP-соединение, предсказывая номера последовательности.
- **RST-атаки:** Отправка поддельных RST-пакетов для принудительного закрытия соединения. Атакующий, зная IP, порты и примерный Sequence Number (или угадывая его), отправляет поддельный RST, и соединение рвётся.
- **Replay Attack:** Повторная отправка ранее перехваченных TCP-сегментов для нарушения целостности данных.

### 2. Инженерный FAQ (На основе разбора дискуссий)

- **Вопрос:** *В каком году впервые был описан TCP протокол?*
- **Технический ответ:** **1981** (RFC 793).

- **Вопрос:** *Какова длина TCP-заголовка без опций?*
- **Технический ответ:** **20 байт**.

- **Вопрос:** *Сколько пакетов нужно для установки TCP-соединения (без учёта ARP)?*
- **Технический ответ:** **3** (SYN → SYN+ACK → ACK).

- **Вопрос:** *Что произойдёт при попытке подключения к закрытому порту?*
- **Технический ответ:** Сервер ответит пакетом с флагом **RST**. Соединение не будет установлено.

- **Вопрос:** *Что такое SACK?*
- **Технический ответ:** Selective Acknowledgment (RFC 2018) — опция TCP, позволяющая подтверждать отдельные фрагменты данных, а не только непрерывную последовательность. Ускоряет восстановление при множественных потерях.

- **Вопрос:** *Почему TCP «зависает» при подключении к недоступному хосту?*
- **Технический ответ:** TCP выполняет повторные попытки с экспоненциально увеличивающимися интервалами (Exponential Backoff), ожидая ответа. Пока не исчерпаны все попытки, приложение ждёт.

## Заключение

TCP — это фундаментальный протокол транспортного уровня, обеспечивающий надёжную, упорядоченную и потоковую передачу данных между приложениями. Его ключевые механизмы (трёхэтапное рукопожатие, номера последовательности и подтверждения, окна перегрузки и повторная передача) гарантируют доставку данных в условиях нестабильных сетей. Понимание состояний TCP, структуры заголовка и поведения при различных сценариях (закрытый порт, недоступный хост) является обязательным для любого сетевого инженера и специалиста по информационной безопасности.
