# Тема 4.1: Транспортный уровень

Изучение роли транспортного уровня в модели OSI, его ключевых протоколов (TCP и UDP), концепции портов для идентификации приложений, а также практических инструментов диагностики (`netstat`, `ss`).

---

## 1. Транспортный уровень — Базовые свойства

- **Уровень OSI:** Транспортный уровень (L4).
- **Сущность данных (PDU):** Сегмент (для TCP) / Датаграмма (для UDP).
- **Принцип работы:** Транспортный уровень обеспечивает доставку данных между **приложениями** на хостах.

### Иерархия доставки

| **Уровень** | **Что обеспечивает** |
| :--- | :--- |
| Канальный уровень (L2) | Доставку данных от одного хоста к другому в рамках **одного сетевого сегмента** |
| Сетевой уровень (L3) | Доставку данных между **хостами** (в одном или разных сегментах) |
| **Транспортный уровень (L4)** | Доставку данных между **приложениями** на хостах |
*Таб. 1. Иерархия доставки.*

### Протоколы транспортного уровня

В стеке TCP/IP на транспортном уровне работают два основных протокола:

| **Протокол**                            | **Особенности**                                                                                                                    |
| :-------------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------- |
| **UDP** (User Datagram Protocol)        | Протокол **без гарантии** доставки данных. Если пакет потеряется, UDP ничего не будет делать. Не гарантирует даже отправку пакета. |
| **TCP** (Transmission Control Protocol) | Протокол с **гарантией** доставки данных. Обнаруживает и восстанавливает потерянные пакеты. Заботится о скорости передачи данных.  |
*Таб. 2. Протоколы транспортного уровня и их особенности.*

> **Шутка для запоминания:** «Я знаю отличную шутку про UDP, но не факт, что она до вас дойдёт. А ещё я знаю отличную шутку про TCP, но если она до вас не дойдёт, то я повторю».

## 2. Порты — идентификаторы приложений

- **Определение:** Порт — это поле размером **2 байта (16 бит)**.
- **Количество портов:** $2^{16} = 65 536$ (от `0` до `65535`).
- **Назначение:** Порт уникально идентифицирует приложение на хосте. Один порт не может быть открыт разными приложениями **в одно время**. Одно приложение может открыть несколько портов.

**Важно:** Пара **`IP-адрес:Порт`** уникально идентифицирует приложение в глобальной сети Интернет.

### Диапазоны портов

| **Диапазон** | **Название** | **Особенности** |
| :--- | :--- | :--- |
| `0` – `1023` | **Привилегированные** (Well-Known) | Может открыть только администратор/суперпользователь. На этих портах работают известные службы (HTTP: 80, HTTPS: 443, DNS: 53, SMTP: 25). |
| `1024` – `65535` | **Пользовательские** (Registered / Dynamic) | Может открыть обычный пользователь. |
*Таб. 3. Диапазоны портов.*

> **Пример:** Веб-сервер (HTTP) по умолчанию открывает порт `80`. Безопасный веб-сервер (HTTPS) — порт `443`.

**Официальный список портов:** [IANA Service Name and Port Number Registry](https://www.iana.org/assignments/service-names-port-numbers/service-names-port-numbers.xhtml)

## 3. Практическая диагностика: netstat и ss

Для просмотра открытых и активных портов используется утилита **`netstat`** (или более современная **`ss`**).

### Команды для разных ОС

| **ОС**      | **Команда**                            |
| :---------- | :------------------------------------- |
| **Windows** | `netstat -ano` (или `netstat -an`)     |
| **Linux**   | `netstat -anltp` (или `sudo ss -tuap`) |
| **MacOS**   | `netstat -anl`                         |
*Таб. 4. Утилиты для просмотра портов в разных ОС.*

### Колонки вывода netstat

| **Колонка**         | **Описание**                                                                                           |
| :------------------ | :----------------------------------------------------------------------------------------------------- |
| **Proto**           | Протокол транспортного уровня: `tcp`, `udp`, `tcp6` (IPv6).                                            |
| **Local Address**   | Локальный IP-адрес и порт. Если `0.0.0.0` — приложение слушает все IP-адреса хоста.                    |
| **Foreign Address** | Удалённый IP-адрес и порт (адрес другого приложения).                                                  |
| **State**           | Состояние соединения (только для TCP). UDP не имеет состояний.                                         |
| **PID** (Linux)     | Идентификатор процесса. При запуске от суперпользователя показывает имя программы (например, `nginx`). |
*Таб. 5. Описание колонок вывода команды netstat.*

> **Пример:** `0.0.0.0:80` → Веб-сервер **nginx** открыл порт 80 и готов обслуживать запросы.

### Альтернатива netstat: ss

В современных дистрибутивах Linux рекомендуется использовать **`ss`**:
```bash
ss -tuap
```

## 4. Разбор сложных инженерных ситуаций и аномалий

### 4.1. Состояния UDP в netstat

**Вопрос:** *Почему у UDP иногда отображаются состояния (ESTABLISHED) в netstat/ss?*

**Технический ответ:** Протокол UDP действительно не имеет состояний, так как он не устанавливает соединение. Однако некоторые реализации (например, в Linux) могут показывать состояние `UNCONN` (по умолчанию) или даже `ESTABLISHED`, если сокет был привязан к удалённому адресу с помощью `connect()`. Это не влияет на работу протокола — UDP остаётся протоколом без установки соединения.

### 4.2. Могут ли несколько приложений использовать один порт?

**Вопрос:** *Можно ли открыть один порт разными приложениями?*

**Технический ответ:** **Нет, в одно и то же время**. Однако одно приложение может открыть несколько портов, а разные приложения могут использовать один и тот же порт **в разное время** (например, поочерёдный запуск веб-серверов на 80-м порту).

### 4.3. Что такое Foreign Address = 0.0.0.0:*?

**Вопрос:** *Что означает `0.0.0.0:*` в колонке Foreign Address?*

**Технический ответ:** Это означает, что приложение готово принимать входящие подключения от **любого** удалённого хоста и с **любого** порта. Такая запись характерна для серверов, находящихся в состоянии `LISTEN`.

### 4.4. Что будет, если портов не хватит?

**Вопрос:** *А если запущенных приложений будет больше 65 535, и для каждого нужен свой порт?*

**Технический ответ:** Кому-то просто не хватит портов. На практике такое случается крайне редко, так как:
- Одно приложение может использовать несколько портов, но 65 тысяч одновременных сетевых приложений на одном хосте — это экзотика.
- В реальных системах общее количество процессов с сетевым взаимодействием обычно не превышает нескольких тысяч.

### 4.5. Особенности флагов netstat и ss

**Вопрос:** *В чём разница между флагами `-a` и `-l` в `netstat/ss`?*

**Технический ответ:**
- **`-a` (all)** — показывает **все** сокеты: слушающие порты (`LISTEN`), установленные соединения (`ESTABLISHED`), а также порты в состоянии `TIME_WAIT`.
- **`-l` (listening)** — показывает **только слушающие** сокеты (те, что ждут входящих подключений).

**Важный нюанс:** Флаг `-l` **не перекрывает** `-a`. Эти флаги не являются взаимоисключающими, они работают как фильтры поверх общего набора данных.

**Практический совет:** На практике для быстрого просмотра слушающих портов чаще используют связку:

```bash
netstat -tuln
```

или

```bash
ss -tuln
```

Где:

- `-t` — TCP
- `-u` — UDP
- `-l` — только слушающие
- `-n` — числовые адреса (без разрешения DNS-имён)

Флаг `-a` в такой связке обычно **не ставят**, потому что он добавляет лишнюю информацию (установленные соединения), которая в этом контексте не нужна.

## 5. Расширенный ИБ-контекст и инженерный FAQ

### 1. Векторы атак и уязвимости

- **Сканирование портов (Port Scanning):** Злоумышленники используют утилиты типа `nmap` для поиска открытых портов и уязвимых служб.
- **Перехват портов (Port Hijacking):** Если приложение закрыло порт, злоумышленник может открыть его и перехватывать трафик.
- **DDoS-атаки на порты:** Перегрузка определённого порта (например, 80/443) может вывести веб-сервер из строя.
- **Уязвимости привилегированных портов:** Если злоумышленник получит доступ к административным правам, он сможет открыть привилегированные порты (`0-1023`) и маскировать свои службы под легитимные.

### 2. Инженерный FAQ (На основе разбора дискуссий)

- **Вопрос:** *Зачем вообще применять UDP, если TCP надёжнее?*
- **Технический ответ:** UDP используется там, где **скорость** важнее надёжности: потоковое видео, онлайн-игры, VoIP, DNS-запросы. TCP добавляет задержки на установку соединения и повторную передачу данных.

- **Вопрос:** *В чём разница между `netstat` и `ss`?*
- **Технический ответ:** `ss` — более современная и быстрая утилита, пришедшая на смену `netstat`. Она показывает больше информации о сокетах и работает быстрее. Рекомендуется к использованию в новых дистрибутивах Linux.

- **Вопрос:** *Что используется для адресации на транспортном уровне?*
- **Технический ответ:** **Порты**.

- **Вопрос:** *Сколько всего портов у TCP или UDP?*
- **Технический ответ:** **65 536** (от `0` до `65535`).

- **Вопрос:** *Как в Windows узнать, какое приложение открыло порт?*
- **Технический ответ:** Используйте `netstat -ano` для получения PID, затем найдите процесс в Диспетчере задач. В PowerShell можно использовать: `Get-NetTcpConnection -State Listen | Select-Object LocalAddress, LocalPort, OwningProcess`. Также есть графический инструмент **TCPView** от Microsoft Sysinternals.

## Заключение

Транспортный уровень является ключевым звеном между приложениями и сетью. Он обеспечивает адресацию приложений через порты и предоставляет два основных протокола: TCP (надёжный, но медленный) и UDP (ненадёжный, но быстрый). Понимание портов, их диапазонов и принципов работы диагностических утилит (`netstat`, `ss`) является фундаментальным навыком для любого сетевого инженера и специалиста по информационной безопасности.
