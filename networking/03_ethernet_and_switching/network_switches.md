# Тема 2.5: Коммутатор (Свитч)

Изучение принципов работы активного сетевого оборудования канального уровня, детального алгоритма заполнения таблиц коммутации (включая аномальные ситуации), механизмов разделения сред передачи данных и угроз информационной безопасности в L2-сегментах.

---

## 1. Коммутатор (Свитч) — Базовые свойства

Для решения проблем с масштабируемостью и производительностью сети, которые были присущи концентраторам (хабам), был предложен свитч.

- **Уровень OSI:** Работает на **втором (канальном) уровне (L2)** модели ISO/OSI. Это значит, что он умеет работать с Ethernet-кадрами.
- **Внешний вид:** Выглядит коммутатор так же, как и хаб — небольшая коробка с портами для подключения патч-кордов.
- **Принцип работы:** Свитч — это «поумневший хаб». Он пытается отправлять Ethernet-пакеты (кадры) строго в тот порт, куда подключен хост назначения, вместо того чтобы забивать мусорным трафиком всю сеть.

---

## 2. Уникальность адресов и анатомия таблицы коммутации (CAM)

На канальном уровне все хосты используют **MAC-адрес** (физический адрес устройства). Он обязан быть уникальным в рамках одного сегмента сети. Если в сети появятся два хоста с одинаковым MAC-адресом, однозначно доставить кадр получателю станет невозможно, и сеть начнет работать со сбоями.

Для адресной отправки свитч использует внутреннюю **таблицу коммутации** (CAM-таблицу), в которой хранится информация о том, за каким физическим портом коммутатора закреплен конкретный MAC-адрес.

### Фундаментальное правило работы свитча:

> Коммутатор обучается и заполняет таблицу коммутации исключительно на основе MAC-адреса **отправителя** (Source MAC) входящего кадра. MAC-адрес получателя (Destination MAC) используется только для принятия решения о направлении пересылки, но **никогда** в таблицу не заносится.

### Подробный пошаговый алгоритм работы коммутатора

Рассмотрим сеть из трех хостов, построенную через свитч:

- `host_1` (MAC: `00:00:00:00:00:01`) подключен в **Порт 1**
- `host_2` (MAC: `00:00:00:00:00:02`) подключен in **Порт 2**
- `host_3` (MAC: `00:00:00:00:00:03`) подключен в **Порт 3**

#### Шаг 1: Исходное состояние (Включение питания)

Когда коммутатор только включается, его таблица коммутации абсолютно пуста. Он еще ничего не знает об окружающей топологии.

| **MAC-адрес отправителя** | **Порт свитча** |
| ------------------------- | --------------- |
| -                         | -               |
| -                         | -               |
_Таб. 2. Исходная пустая таблица коммутации_

#### Шаг 2: Первое отправление (`host_1` шлет кадр в сторону `host_2`)

1. Кадр от `host_1` приходит на **Порт 1** коммутатора.
2. Коммутатор смотрит на поле **Source MAC** (`00:00:00:00:00:01`) и порт прибытия (**1**). Он делает первую запись в свою таблицу.
3. Затем коммутатор смотрит на поле **Destination MAC** (`00:00:00:00:00:02`) и ищет его в таблице.
4. Так как таблица пуста, запись для `host_2` отсутствует. Возникает ситуация **Unknown Unicast** (неизвестный unicast-адрес).
5. Коммутатор вынужден сдублировать этот кадр во **все активные порты**, кроме того, откуда он пришел (то есть отправляет в Порт 2 и Порт 3). На этом этапе он ведет себя как обычный хаб.

| **MAC-адрес отправителя** | **Порт свитча** |
| ------------------------- | --------------- |
| `00:00:00:00:00:01`       | 1               |
| -                         | -               |
_Таб. 3. Таблица коммутации после первого шага с одной записью_

#### Шаг 3: Ответная реакция (`host_2` отвечает хосту `host_1`)

1. `host_2` получает кадр, видит, что он предназначен ему, и отправляет ответный кадр в сторону `host_1`.
2. Кадр от `host_2` поступает на **Порт 2** коммутатора.
3. Коммутатор считывает **Source MAC** (`00:00:00:00:00:02`) и заносит его в таблицу напротив **Порта 2**.
4. Далее коммутатор смотрит на **Destination MAC** (`00:00:00:00:00:01`). Он ищет его в таблице и **находит** запись: данный адрес находится за Портом 1.
5. Коммутатор отправляет кадр **строго направленно (адресно)** в Порт 1. `host_3` (Порт 3) этот трафик уже не получает. Сеть начинает работать эффективно.

| **MAC-адрес отправителя** | **Порт свитча** |
| ------------------------- | --------------- |
| `00:00:00:00:00:01`       | 1               |
| `00:00:00:00:00:02`       | 2               |
_Таб. 4. Обученная таблица коммутации_

## 3. Разбор сложных инженерных ситуаций и аномалий

### Ситуация А: Несколько MAC-адресов на одном порту

Если к одному порту коммутатора подключить хаб (или другой свитч), к которому, в свою очередь, подключено несколько компьютеров, то коммутатор успешно запомнит их все. В CAM-таблице напротив одного и того же номера порта будет записано несколько строчек с разными MAC-адресами.

| **MAC-адрес отправителя** | **Порт свитча** |
| ------------------------- | --------------- |
| `00:00:00:00:00:01`       | 1               |
| `00:00:00:00:00:02`       | 1               |
| `00:00:00:00:00:03`       | 1               |
_Таб. 5. Пример таблицы при подключении разветвителя на Порт 1_

### Ситуация Б: Перемещение устройства между портами (MAC Roaming)

Если переткнуть кабель устройства из Порта 1 в Порт 2, коммутатор обновит данные автоматически при первой же отправке пакета с этого устройства. Он увидит знакомый Source MAC на новом порту, сотрет старую запись для Порта 1 и создаст актуальную для Порта 2.

### Ситуация В: Физические петли и кольца в Ethernet

Если случайно (или намеренно) соединить порты коммутаторов кабелем в кольцо, сеть мгновенно выйдет из строя. На канальном уровне Ethernet **нет механизма времени жизни пакета (в отличие от поля TTL в IP-пакетах)**.

Широковещательный кадр, попав в такое кольцо, начнет бесконечно циркулировать по кругу, размножаясь на каждом порту. Это приводит к **широковещательному шторму (Broadcast Storm)**. Свитч тратит все ресурсы процессора и памяти на обработку этих копий, буферы переполняются, и легитимная сеть полностью перестает функционировать. Для борьбы с петлями используется специальный протокол **STP (Spanning Tree Protocol)**.

### Ситуация Г: Существование коллизий и затирание MAC-адресов

- **Домен коллизий:** Свитч делит сеть на изолированные домены коллизий. Каждый порт коммутатора — это отдельный домен коллизий.
- **Полный дуплекс:** При использовании полнодуплексного режима (Full-Duplex) на линии «карта-порт» коллизии исчезают физически, так как каналы приема и передачи разделены.
- **Важный нюанс (Аномалия):** Если на порту свитча всё же возникнет физическая коллизия (например, при работе в Half-Duplex через хаб), или если в симуляторе Miminet смоделировать ошибку (задать хостам одинаковые MAC-адреса), то коммутатор начнет **затирать (перезаписывать)** старые записи в CAM-таблице новыми данными. Постоянная перезапись таблицы приводит к нестабильности — пакеты начинают уходить не туда или свитч сваливается в Unknown Unicast (Flood трафика).

### Ситуация Д: Как устроен домашний Wi-Fi роутер?

Желтые порты (LAN) на задней панели обычного домашнего роутера — это не маршрутизатор. Конструктивно это самый обычный **коммутатор (свитч)** канального уровня, который физически и логически встроен в один корпус с беспроводной точкой доступа и L3-маршрутизатором.

## 4. Домен широковещания (Broadcast Domain)

**Широковещательный домен** — это вся совокупность устройств, до которых дойдет кадр, отправленный на широковещательный MAC-адрес назначения **`FF:FF:FF:FF:FF:FF`**.

- Свитчи изолируют домены коллизий, но они **не изолируют широковещательный трафик**. Если хост отправляет широковещательный пакет (например, ARP-запрос), свитч обязан послушно размножить его и отправить во все свои порты.
- Вся сеть, построенная на свитчах и хабах в рамках одной подсети, представляет собой **один большой широковещательный домен**. Разделить его на части могут только устройства L3-уровня (маршрутизаторы/роутеры) или технологии VLAN.

## 5. Расширенный ИБ-контекст (Анализ угроз и разбор вопросов)

Принципы построения CAM-таблиц лежат в основе классических векторов атак на канальном уровне:

1. **Атака CAM Table Overflow (Переполнение таблицы коммутации):**
   Память коммутатора под CAM-таблицу жестко ограничена железными спецификациями. Злоумышленник запускает утилиту (например, `macof`), которая генерирует десятки тысяч кадров в секунду с фальшивыми случайными Source MAC-адресами. Таблица мгновенно забивается мусором, вытесняя легитимные адреса.
   
   **Результат:** Любой новый легитимный кадр становится для свитча *Unknown Unicast*. Свитч деградирует до состояния хаба и начинает транслировать вообще весь трафик во все порты. Атакующий включает Promiscuous Mode на своей сетевой карте и спокойно перехватывает чужие конфиденциальные данные (пассивный сниффинг).

2. **Атака MAC Spoofing (Подмена адреса):**
   Атакующий отправляет кадры, подделывая Source MAC под адрес легитимной жертвы (или шлюза). Свитч перенаправляет порт жертвы на порт атакующего. В сети начинается процесс *MAC Flapping* (постоянные прыжки адреса между портами), вызывающий отказ в обслуживании (DoS) для легитимного пользователя и перехват его пакетов.

3. **Разбор критической ситуации: Аномальный кадр с Source MAC = Broadcast**
   *Вопрос с форума инженеров:* Что произойдет, если на свитч придет кадр, у которого в поле **Source MAC (адрес отправителя)** будет стоять широковещательный адрес `FF:FF:FF:FF:FF:FF`? Запишет ли свитч его в таблицу на первый порт?
   
   - **Технический ответ:** **Нет, не запишет.** Такого MAC-адреса в принципе не может существовать на сетевой карте физического хоста. По стандарту широковещательный адрес `FF:FF:FF:FF:FF:FF` может выступать **только в роли адреса назначения (Destination MAC)**.
   - **Поведение свитча:** Коммутатор распознает такой кадр как критическую протокольную аномалию (нарушение стандартов Ethernet). Из соображений безопасности и стабильности логики сети, коммутатор **мгновенно отбросит (уничтожит) такой пакет**. Никакие изменения в CAM-таблицу внесены не будут.
