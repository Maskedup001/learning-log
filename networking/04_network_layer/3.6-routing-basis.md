## 2. Проблема масштабирования и появление маршрута по умолчанию

Добавим третий сегмент сети и новую IP-сеть `192.168.1.0/24`. Чтобы хост 1 мог отправлять пакеты на хост 4, необходимо добавить еще одну запись в таблицу маршрутизации:

*(Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=6547afac-26fc-474e-8e4e-9446992ab96c))*

### Расширенная таблица маршрутизации

| IP сеть назначения | IP адрес маршрутизатора |
| :--- | :--- |
| `10.0.0.0/24` | `10.0.0.1` |
| `172.16.0.0/24` | `10.0.0.100` |
| `192.168.1.0/24` | `10.0.0.100` |
*Таб. 3. Таблица маршрутизации для хоста 1 с использованием маски подсети.*

Каждый раз при появлении новой IP-сети за маршрутизатором 1 нам нужно добавлять запись в таблицу маршрутизации, где адресом маршрутизатора всегда будет `10.0.0.100`. Это неудобно и тормозит масштабирование.

**Решение:** Для всех неизвестных IP-сетей добавляется запись `0.0.0.0/0` — **маршрут по умолчанию**. Она означает: "для всех остальных IP-сетей, кроме моей, идти на маршрутизатор".

### Принцип работы:
- Любой IP-адрес при наложении маски `0.0.0.0` даёт `0.0.0.0`
- Таблица маршрутизации просматривается **от большей маски к меньшей**: сначала `/32`, потом `/31`, ... и в конце `/0`

### Финальная таблица с маршрутом по умолчанию

| IP сеть назначения | IP адрес маршрутизатора |
| :--- | :--- |
| `10.0.0.0/24` | `10.0.0.1` |
| `0.0.0.0/0` | `10.0.0.100` |
*Таб. 4. Таблица маршрутизации с маршрутом по умолчанию для хоста 1.*

> **Важное терминологическое уточнение:**  
> Запись `0.0.0.0/0` — это **маршрут по умолчанию**. А **Default Gateway (шлюз по умолчанию)** — это IP-адрес маршрутизатора, на который направляется этот маршрут (в примере — `10.0.0.100`). Это различие критично на собеседованиях и в реальной работе сетевого инженера.

## 3. Таблицы маршрутизации в операционных системах

У каждого хоста, подключенного к сети, есть таблица маршрутизации. Именно по ней хост принимает решения: доставлять пакет самостоятельно или отправлять на маршрутизатор.

### Windows — команда `route print`

Ключевые записи в выводе:
- **`0.0.0.0`** с маской `0.0.0.0` и адресом шлюза `192.168.0.1` — **маршрут по умолчанию**
- **`192.168.0.0`** с маской `255.255.255.0` и адресом шлюза `192.168.0.107` (On-link) — маршрут до своей сети, где `On-link` означает, что хосты находятся в одном широковещательном домене и общаются напрямую по MAC-адресам, без участия маршрутизатора
- **Метрика** — используется для выбора приоритетного маршрута, если есть несколько одинаковых записей. Чем **меньше** метрика, тем **выше** приоритет. Используется для отказоустойчивости.

### Linux / MacOS — команды `netstat -nr` или `ip route`

Ключевые записи:
- **`0.0.0.0`** с маской `0.0.0.0` и шлюзом `10.129.0.1` — маршрут по умолчанию
- **`10.129.0.0`** с маской `255.255.255.0` и шлюзом `0.0.0.0` — маршрут до своей сети (шлюз `0.0.0.0` означает, что хост сам знает, как общаться с этой сетью, т.е. это аналог `On-link` в Windows)

> **Примечание:** В современных дистрибутивах Linux утилита `netstat` может отсутствовать. Используйте команду `ip route` для просмотра таблицы маршрутизации.

## 4. Default-Free Zone (DFZ) — зона без маршрутов по умолчанию

Маршрут по умолчанию удобен, но он лишь **перекладывает проблему** на следующий маршрутизатор. Рано или поздно должен появиться маршрутизатор, который знает маршруты **до всех сетей** — это и есть **Default-Free Zone (DFZ)**.

*(Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=bcbdb588-21a0-4c28-b84f-f03edf06ae29))*

Рассмотрим сеть из 4 сегментов:
- `10.0.0.0/24` — Хост 1 и Маршрутизатор 1
- `192.168.1.0/24` — Маршрутизатор 1 и Маршрутизатор 2
- `172.16.12.0/24` — Маршрутизатор 2 и Маршрутизатор 3
- `169.254.1.0/24` — Маршрутизатор 3 и Хост 2

Если на Маршрутизаторе 2 настроить маршрут по умолчанию на Маршрутизатор 3, то пакет от Хоста 1 до Хоста 2 **дойдёт**, но обратно **не вернётся**, так как Маршрутизатор 2 не знает про сеть `10.0.0.0/24`.

### Решение: DFZ

Чтобы сеть работала, нужно:
1. **Либо** добавить на Маршрутизатор 2 маршруты до **обеих** сетей (`10.0.0.0/24` и `169.254.1.0/24`) — тогда в DFZ будет только Маршрутизатор 2
2. **Либо** добавить маршрут по умолчанию и один из маршрутов — тогда в DFZ будет Маршрутизатор 2 и Маршрутизатор 1 (или 3)

*(Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=8de49b2d-563c-4519-9119-41be6939dd7b))*

### Таблица маршрутизации Маршрутизатора 2 (DFZ)

| IP сеть назначения | IP адрес маршрутизатора |
| :--- | :--- |
| `192.168.1.0/24` | `192.168.1.102` (сам Маршрутизатор 2) |
| `172.16.12.0/24` | `172.16.12.102` (сам Маршрутизатор 2) |
| `10.0.0.0/24` | `192.168.1.101` (на Маршрутизатор 1) |
| `169.254.1.0/24` | `172.16.12.103` (на Маршрутизатор 3) |
*Таб. 5. Таблица маршрутизации Маршрутизатора 2 (DFZ).*

## 5. Один сетевой сегмент, разные IP-сети

Рассмотрим сеть из 3 хостов, подключенных к одному свитчу:
- Хост 1: `192.168.1.1/24`
- Хост 2: `192.168.1.2/24`
- Хост 3: `192.168.0.3/24`

Хосты 1 и 2 находятся в одной IP-сети (`192.168.1.0/24`), а хост 3 — в другой (`192.168.0.0/24`). Хотя физически и канально они могут обмениваться пакетами (через свитч), на сетевом уровне они "не видят" друг друга.

*(Эмулятор: [miminet.ru](https://miminet.ru/web_network?guid=adfb3817-4161-4c82-9578-1934e70f8a41))*

### Таблица маршрутизации Хоста 1

| IP сеть назначения | IP адрес маршрутизатора |
| :--- | :--- |
| `192.168.1.0/24` | `192.168.1.1` |
*Таб. 6. Таблица маршрутизации для хоста 1.*

Эта запись добавляется **автоматически** при настройке IP-адреса и маски хоста. Она означает: "я сам могу общаться со всеми хостами в моей сети".

### Чтобы Хост 1 и Хост 3 могли обмениваться данными:
- Хосту 1 нужно добавить маршрут до сети `192.168.0.0/24`
- Хосту 3 нужно добавить маршрут до сети `192.168.1.0/24`

> **Важно:** В этой конфигурации **свитч не выполняет маршрутизацию** — он просто передаёт кадры между портами. Маршрутизация происходит на уровне IP и требует явного добавления маршрутов в таблицы хостов.

## 6. Разбор сложных инженерных ситуаций и аномалий

- **Default Gateway vs маршрут по умолчанию:** Запись `0.0.0.0/0` — это маршрут по умолчанию. А **Default Gateway** — это IP-адрес маршрутизатора, на который указывает этот маршрут. Не путайте эти понятия — это частый вопрос на собеседованиях.
- **Шлюз `0.0.0.0` в Linux:** В таблицах маршрутизации Linux в поле `Gateway` может стоять `0.0.0.0`. Это означает, что хост сам знает, как доставить пакет в эту сеть (это аналог `On-link` в Windows).
- **Метрика в таблицах маршрутизации:** Используется, когда есть несколько одинаковых маршрутов. Чем **меньше** значение метрики, тем **выше** приоритет. Это позволяет реализовать отказоустойчивость (например, два маршрута по умолчанию с разными метриками).
- **Роль свитча при разных IP-сетях:** Свитч работает на канальном уровне (L2) и не "знает" про IP-адреса. Он может передавать кадры между хостами из разных IP-сетей, но для обмена данными на сетевом уровне (L3) требуется явная маршрутизация (добавление записей в таблицы маршрутизации).
- **Loopback (127.0.0.0/8):** Пакеты, отправленные на любой адрес из этого диапазона, никогда не покидают хост. Поэтому подключиться к другу по адресу `127.0.0.1` невозможно — этот адрес всегда указывает на самого себя.

## 7. Расширенный ИБ-контекст и инженерный FAQ

### 1. Векторы атак и уязвимости

- **Подмена маршрута по умолчанию (Default Gateway Spoofing):** Атакующий в локальной сети может подменить ARP-ответы и выдать свой MAC-адрес как MAC-адрес шлюза по умолчанию. Весь трафик жертвы будет идти через атакующего (`MITM`).
- **Неправильная статическая маршрутизация:** Ошибки в настройке статических маршрутов могут привести к **петлям маршрутизации** (routing loops) и недоступности сетей.
- **Сетевые петли (Loopback):** Понимание того, что `127.0.0.0/8` никогда не покидает хост, критично для безопасности — это предотвращает попытки внешнего доступа к локальным сервисам через этот адрес.

### 2. Инженерный FAQ (На основе разбора дискуссий)

- **Вопрос:** *Чем хост отличается от маршрутизатора?*  
  **Технический ответ:** Ключевое отличие — **маршрутизатор пересылает чужие пакеты**, а хост — только свои (сгенерированные им самим). Однако, если на хосте включен параметр `IP forwarding` (в Linux — `net.ipv4.ip_forward=1`), он может выполнять функции маршрутизатора. Также в отдельном случае маршрутизатор может выступать в роли хоста, если к нему подключаются по SSH для управления. В общем случае, хостом называют любое конечное устройство с IP-адресом (ПК, сервер, телефон, IoT-устройство).

- **Вопрос:** *Таблица маршрутизации есть у всех устройств в сети?*  
  **Технический ответ:** Нет, таблица маршрутизации есть **у всех хостов** (устройств с IP-адресом, которые могут генерировать и принимать пакеты). У L2-коммутаторов (свитчей) и концентраторов (хабов) таблицы маршрутизации нет, так как они работают на канальном уровне и не принимают решений об IP-маршрутизации.

- **Вопрос:** *Что означает `On-link` в таблице маршрутизации Windows?*  
  **Технический ответ:** Запись `On-link` означает, что сетевые устройства находятся в одном широковещательном домене (локальной сети) и могут обмениваться данными **напрямую**, используя MAC-адреса, без участия маршрутизатора. Это аналог шлюза `0.0.0.0` в таблицах Linux.

- **Вопрос:** *Сможет ли хост 1 и хост 3 из примера в разделе 5 обменяться данными без добавления маршрутов?*  
  **Технический ответ:** Нет. Хотя они физически подключены к одному свитчу и могут общаться на канальном уровне (L2), на сетевом уровне (L3) они находятся в разных IP-сетях. Хост 1 будет считать, что хост 3 находится в другой сети, и попытается отправить пакет на шлюз по умолчанию. Если шлюз не настроен, связь будет невозможна. Для прямого общения необходимо добавить маршруты в таблицы маршрутизации обоих хостов.

## Заключение

Маршрутизация является ключевым механизмом сетевого уровня, позволяющим устройствам принимать решения о пересылке пакетов. Эволюция от статических поузловых записей к маршруту по умолчанию (`0.0.0.0/0`) значительно упростила масштабирование сетей. Однако в сложных topologies, где маршрутов по умолчанию недостаточно, необходима зона без маршрутов по умолчанию (`DFZ`), содержащая все маршруты до всех сетей. Понимание поведения таблиц маршрутизации в различных ОС, различие между хостом и маршрутизатором, а также грамотное использование метрик — фундаментальные навыки сетевого инженера и специалиста по информационной безопасности.
