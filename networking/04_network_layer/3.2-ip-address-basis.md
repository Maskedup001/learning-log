# Тема 3.2: IP адреса

Изучение базового формата IPv4-адресации, команд проверки сетевых интерфейсов в различных операционных системах, а также фундаментальной проблемы инкапсуляции при сопоставлении логических L3-адресов с физическими L2-идентификаторами.

---

## 1. Протокол IP и формат адреса — Базовые свойства

- **Уровень OSI:** L3 (Сетевой уровень / Network Layer).
- **Сущность данных (PDU):** Пакет (Packet).
- **Принцип работы:** На сетевом уровне протокол `IP` (`Internet Protocol`), описанный в стандарте `RFC 791`, использует уникальные глобальные идентификаторы — `IP` адреса — для сквозной маршрутизации и адресации хостов в рамках всей сети Интернет.

---

## 2. Технический алгоритм / Анатомия процесса по шагам

### Структура адреса и бинарная логика

`IP` адрес представляет собой единое поле размером в `4 байта` (`32 бита`), разделенное для удобства восприятия на 4 группы десятичных чисел (октетов) через точку. 
Каждое число жестко соответствует `8 битам` адресного пространства. Максимальное бинарное значение одного октета равно `11111111` в двоичной системе, что в десятичном представлении равняется `255`. Физическое добавление 9-го бита невозможно, поэтому ни одно число в `IP` адресе не может превышать `255`. В рамках одного октета доступно ровно 256 значений: в диапазоне от `0` до `255`.

### Проверка адреса на хосте (Диагностический алгоритм)

В зависимости от целевой ОС, получение информации об установленном `IPv4` адресе выполняется через интерфейс командной строки:

1. **В ОС Windows:** Вызов команды `ipconfig` или `ipconfig /all` $\rightarrow$ поиск активного интерфейса (например, «Беспроводная сеть») $\rightarrow$ чтение строки «IPv4-адрес» (например, `192.168.0.107`).
2. **В ОС Linux / MacOS:** Вызов команды `ifconfig` или современного аналога `ip a` (для Linux) $\rightarrow$ локализация интерфейса (например, `en0` на Mac) $\rightarrow$ чтение значения в поле `inet`. Специфический метод для MacOS без ручного поиска интерфейса: команда `networksetup -getinfo Wi-Fi`.

## 3. Разбор сложных инженерных ситуаций и аномалий

### Аномалия L2-инкапсуляции: Проблема отсутствующего MAC-адреса получателя

Рассматривается базовая топология: два хоста соединены через один коммутатор (L2-свитч).
- `хост 1` — `10.0.0.1`
- `хост 2` — `10.0.0.2`

*(Эмулятор для тестов: [miminet.ru](https://miminet.ru/web_network?guid=d9012b77-fd52-48b0-b618-a3c3f55caf15))*

При инициации отправки пакета утилитой `ping` (отправка `ICMP Echo Request`) от `хоста 1` к `хосту 2`, стек протоколов сталкивается с критической нехваткой данных на канальном уровне:

- **На сетевом уровне (L3):** Заголовки заполняются без ошибок. `Source IP` берется из параметров собственного интерфейса (`10.0.0.1`), а `Destination IP` извлекается из аргументов команды выполнения (`10.0.0.2`).
- **На канальном уровне (L2):** Формируется кадр. Поле `Source MAC` успешно заполняется физическим адресом сетевой карты `хоста 1`. Поле `Destination MAC` **не может остаться пустым**, так как сетевые интерфейсы оборудования не смогут обработать кадр без адреса назначения. Возникает состояние тупика: `хост 1` на текущем этапе знает логический адрес (`IP`), но не знает физический адрес (`MAC`) получателя.

---

## 4. Границы доменов и лимиты масштабирования

- **Масштабирование IP:** Поскольку `IP` адрес является уникальным глобальным логическим идентификатором, он позволяет абстрагироваться от локальных топологий, выступая базой для построения глобальной сети Интернет.
- **Лимит изоляции L2:** Локальный свитч коммутирует кадры исключительно на основании `MAC` адресов. Но он не может самостоятельно транслировать `IP` в `MAC`. Для предотвращения лавинообразных сбоев на этапе отправки `IP` пакета, хост обязан локально решить задачу сопоставления адресов до отправки кадра в физическую среду.

## 5. Расширенный ИБ-контекст и инженерный FAQ

### 1. Векторы атак и уязвимости

- **Разведка сети (ICMP Scanning):** Утилита `ping` напрямую используется злоумышленниками для первичного сканирования пространства `IP` адресов целевой сети с целью выявления активных хостов по их `ICMP` ответам.
- **Манипуляции с ARP-кэшем:** Сама логическая задержка, возникающая на этапе «Какой `MAC` адрес получателя?», вынуждает операционную систему задействовать механизмы динамического сопоставления адресов, которые уязвимы к атакам класса `ARP Spoofing` (будет рассмотрено в следующих темах).

### 2. Инженерный FAQ

- **Вопрос:** *Историческая справка: стандарты RFC по IP вышли в 1981 году, а по ARP — только в 1982 году. Как сисадмины первых сетей решали задачу отправки пакетов, если механизмы автоматического определения MAC-адреса назначения отсутствовали?*
- **Технический ответ:** В ранних сетях сопоставление выполнялось исключительно в ручном режиме. Сетевые инженеры жестко прописывали статическую таблицу соответствия логических `IP` и физических `MAC` адресов в специальный конфигурационный файл операционной системы, который выполнял роль статического `ARP` кэша. С ростом инфраструктуры ручная поддержка файлов конфигурации на каждом узле стала неэффективной, что привело к разработке автоматизированных протоколов динамического разрешения адресов.
- **Вопрос:** *Зачем вообще нужен MAC-адрес, если IP-адрес уже объявлен как уникальный идентификатор в рамках всей сети Интернет? Или наоборот — зачем IP, если есть MAC?*
- **Технический ответ:** Это фундаментальный принцип разделения обязанностей в модели OSI. `MAC` адрес — это плоский физический адрес, жестко «зашитый» в оборудование; он необходим для доставки данных внутри одного конкретного локального сегмента (L2). `IP` адрес — это логический, структурированный, иерархический адрес (L3), который не привязан к железу и может динамически меняться при переходе в другую сеть (например, по `DHCP` или при включении `VPN`), в отличие от `MAC`, который остается неизменным. Он необходим для маршрутизации между тысячами разных сетей. Если бы сети строились только на `MAC`, роутеры по всему миру должны были бы хранить таблицы со списками триллионов `MAC` адресов всех существующих девайсов, что технически невозможно. А если бы сети строились только на `IP` без `MAC`, нам пришлось бы аппаратно перестраивать логику работы сетевых чипов и коммутаторов под каждый существующий сетевой протокол (не только `IP`), теряя гибкость и независимость уровней.
- **Вопрос:** *В лекции указано, что команда ifconfig используется повсеместно. Есть ли нюансы для современных дистрибутивов Linux?*
- **Технический ответ:** Да. В современных дистрибутивах Linux команда `ifconfig` признана устаревшей (deprecated) и по умолчанию исключена из базовой поставки системы. Для ее использования требуется ручная установка пакета `net-tools`. Современным индустриальным стандартом в Linux является утилита `ip` (конкретно команда `ip a` или `ip address`), входящая в пакет `iproute2`.

## Сравнение архитектурных идентификаторов L2 и L3

| **Критерий** | **MAC-адрес (L2)** | **IP-адрес (L3)** |
| :--- | :--- | :--- |
| **Размер** | `6 байт` (`48 бит`) | `4 байта` (`32 бита`) |
| **Тип адреса** | Физический (аппаратный) | Логический (программный) |
| **Изменяемость** | Жестко зашит в `NIC` (сетевую карту) | Может динамически меняться (`DHCP`, `VPN`) |
| **Область действия** | В пределах одного сегмента (`LAN`) | Глобально (Интернет) |
| **Уровень OSI** | Канальный (`L2`) | Сетевой (`L3`) |

---

## Заключение

Формат адресации `IPv4` жестко лимитирован разрядностью в 32 бита, что накладывает строгие ограничения на максимальные числовые значения октетов. Наличие уникального логического `IP` адреса получателя не позволяет отправить пакет в физическую среду до тех пор, пока не будет решена задача его сопоставления с локальным `MAC` адресом канального уровня. Без разрешения проблемы сопоставления `IP` $\rightarrow$ `MAC` передача данных в локальной сети физически невозможна, что делает `ARP`-протокол критическим звеном инфраструктуры.
